# Brainberg: Scrapers + Admin + MCP Implementation Plan

## Context

Brainberg is a European AI/tech event aggregation platform (Next.js 16, Drizzle ORM, PostGIS). It has a working frontend (browse, search, filters, detail pages) but no real data — only 12 seed events. We need to build the full data ingestion pipeline from 4 sources (confs.tech, dev.events, Meetup, Eventbrite), admin tools for event management, and an MCP server. Prior analysis confirmed **no AI needed** — all sources provide structured enough data for deterministic processing.

## Phase 1: Schema Migration

**Files modified:** `src/lib/db/schema.ts`, `src/lib/utils.ts`
**Files created:** Migration SQL (generated by Drizzle Kit)

### Changes:
1. **Extend `eventSourceEnum`** — add `confs_tech`, `dev_events`
   - Note: Drizzle Kit can't auto-generate `ALTER TYPE ... ADD VALUE`. We'll hand-write this in migration SQL or use `db:push` for dev.
2. **Extend `eventCategoryEnum`** — add `hacker_maker_community`
   - For hacker camps, maker events, and community congresses (CCC, EMF, eth0, WHY, GPN, MCH, SHA, Maker Faires, etc.)
   - Add to `CATEGORY_LABELS` ("Hacker, Maker & Community") and `CATEGORY_COLORS` in `src/lib/utils.ts`
3. **Add columns to `events`** — `confsTechUrl TEXT`, `devEventsUrl TEXT`
4. **Add `eventSources` junction table** — tracks which sources an event came from
   - `id` UUID PK, `eventId` FK → events, `source` eventSourceEnum, `sourceId` TEXT, `sourceUrl` TEXT, `rawData` JSONB, `firstSeenAt` TIMESTAMPTZ, `lastSeenAt` TIMESTAMPTZ
   - Unique constraint: (eventId, source)
   - Index: (source, sourceId)
5. **Add `eventFingerprints` table** — dedup index
   - `id` UUID PK, `eventId` FK → events, `fingerprintType` TEXT ('url', 'title_date_city'), `fingerprintValue` TEXT
   - Unique constraint: (fingerprintType, fingerprintValue)
6. **Add `scraperRuns` table** — audit trail for scraper executions
   - `id` UUID PK, `source` eventSourceEnum, `status` TEXT ('running','completed','failed'), `eventsFound` INT, `eventsCreated` INT, `eventsUpdated` INT, `eventsDeduplicated` INT, `errorMessage` TEXT, `startedAt` TIMESTAMPTZ, `completedAt` TIMESTAMPTZ
7. **Add relations** for new tables

**Verify:** `pnpm db:generate` produces migration, `pnpm db:push` applies cleanly.

## Phase 2: Shared Utilities

All scrapers share common infrastructure. Create `src/lib/scraper/` directory.

### Files:
| File | Purpose |
|------|---------|
| `src/lib/scraper/types.ts` | `NormalizedEvent` interface, `Scraper` interface (with `scrape(): AsyncGenerator<NormalizedEvent>`) |
| `src/lib/scraper/category-map.ts` | 4 static lookup tables (confs.tech filenames, dev.events RSS categories, Meetup topic urlkeys, Eventbrite category IDs) → our `eventCategoryEnum`. Includes `hacker_maker_community` mappings for hacker camp keywords. Fallback: `general_tech`. Also: AI keyword regex and hacker/maker keyword regex for title-based detection. |
| `src/lib/scraper/city-resolver.ts` | Resolves (cityName, countryName/code) → cityId/countryId from DB. In-memory cache on first call. Alias map for common variants ("Wien"→"Vienna"). If city not found, log + skip. |
| `src/lib/scraper/html-utils.ts` | `stripHtml(html)` → plain text (regex-based, no new deps). `truncate(text, len)` → short description. |
| `src/lib/scraper/url-utils.ts` | `normalizeUrl(url)` — strip www, trailing slash, tracking params, lowercase. Used for Layer 1 dedup. |
| `src/lib/scraper/fingerprint.ts` | `exactFingerprint(title, date, city)` — Layer 2 deterministic hash. `fuzzyMatch(a, b)` — Layer 3 Jaccard word similarity. |
| `src/lib/scraper/dedup.ts` | 3-layer dedup: check URL match → exact fingerprint → fuzzy match. Returns existing event or null. Source priority merge logic. |
| `src/lib/scraper/ingest.ts` | Pipeline: takes AsyncGenerator<NormalizedEvent>, runs dedup, inserts/updates events, creates fingerprints, updates eventSources junction, returns stats. |

### Key design patterns:
- `NormalizedEvent` mirrors the events table shape but with raw cityName/countryCode strings (not IDs) — city-resolver handles the lookup
- Each scraper yields events one at a time (AsyncGenerator) to keep memory low
- `ingest.ts` wraps the full pipeline: resolve city → dedup → insert/update → fingerprint

### Category mapping notes:
- **`hacker_maker_community`** detection via keyword regex on titles: `/\b(CCC|congress|camp|hacker|maker|faire|eth0|emf|electromagnetic field|gpn|gulasch|sha2|mcr|mch|why2|fosdem|hackmeeting)\b/i`
- confs.tech: events in `general.json` matching hacker/maker keywords → `hacker_maker_community`
- dev.events: RSS categories like "Open Source", "Community" + title keyword match → `hacker_maker_community`
- Meetup: Topics like "maker", "hackerspace", "open-source" → `hacker_maker_community`
- Eventbrite: Category for "Hobbies & Special Interest" with tech subcategory + keyword match

## Phase 3: Scrapers

Each implements `Scraper` interface, lives in `src/lib/scraper/sources/`.

### 3.1 confs.tech — `confstech.ts`
- Fetch JSON from `https://raw.githubusercontent.com/tech-conferences/conference-data/main/conferences/2026/{topic}.json` for ~25 topics
- Filter: only European countries (check against countries table)
- Category: filename → lookup table. Override to `ai_ml` if title matches AI keyword regex. Override to `hacker_maker_community` if title matches hacker/maker regex.
- sourceId: hash of `name + startDate`. Fields: title, url, dates, city, country, online. No description/image.

### 3.2 dev.events — `devevents.ts`
- **Pass 1:** Fetch RSS at `https://dev.events/rss.xml`, parse XML
- **Pass 2:** For each event, fetch detail page, extract JSON-LD for full structured data (dates, location, organizer, image, offers)
- Category: RSS `<category>` tags → lookup table. eventType from tags too.
- Rate limit: 1-2 req/sec

### 3.3 Meetup — `meetup.ts`
- Fetch search pages for European tech cities, extract `__NEXT_DATA__` JSON
- Parse Apollo cache for Event objects
- Category: Topic urlkey → lookup table
- Rate limit: 2-3 sec between requests. Start with small city list.

### 3.4 Eventbrite — `eventbrite.ts`
- API v3: `GET /v3/destination/search/?categories=101` with Bearer token
- Expand venue + organizer. Paginate via cursor.
- Category: `category_id` → lookup table
- Skips gracefully if `EVENTBRITE_API_TOKEN` not set. Backoff on 429.

## Implementation Order

```
Phase 1 (Schema) → Phase 2 (Utilities) → Phase 3 (Scrapers) → Phase 4 (Orchestration) → Phase 5 (Admin API) → Phase 6 (Admin UI + MCP)
```

Phases 5-6 depend on Phase 1 (schema) but NOT on Phases 3-4 (scrapers), so admin could be built in parallel if needed.

---

## Key Design Decisions

- **Zero AI** in the pipeline — all deterministic lookup tables + keyword regex
- **3-layer dedup:** URL match → title+date+city fingerprint → fuzzy Jaccard similarity
- **AsyncGenerator scrapers** for low memory usage
- **Source priority merging:** Eventbrite > Meetup > dev.events > confs.tech (richer data wins)
- **ADMIN_SECRET header auth** — simple, temporary, replaced when Auth.js ships
- **Scraper-agnostic ingest pipeline** — same dedup/insert logic for all sources
- **`hacker_maker_community` category** — dedicated category for hacker camps, maker events, community congresses (CCC, EMF, eth0, WHY, GPN, etc.)

## Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| Drizzle Kit can't add enum values | Hand-write `ALTER TYPE` SQL or use `db:push` |
| Meetup blocks scraping | Small city list, delays, degrade gracefully |
| Eventbrite token unavailable | Skip gracefully if not set |
| Dedup false positives | High thresholds, log all decisions for review |
| dev.events RSS format changes | Defensive parsing, skip unparseable entries |
| Hacker/maker events not on major platforms | Curated seed data for known recurring events (CCC, EMF, GPN, etc.) + community submission form later |
